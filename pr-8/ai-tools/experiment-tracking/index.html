    Experiment Tracking¶

    Experiment tracking tools help you log hyperparameters, metrics, and artifacts so you can compare runs, reproduce results, and collaborate with your team. AICR supports Weights & Biases (W&B), MLflow, and TensorBoard.

    Why Track Experiments¶

    Training ML models involves many moving parts: learning rates, architectures, data splits, random seeds. Without systematic tracking, it is easy to lose track of which configuration produced which result.

    Experiment tracking tools solve this by:

    • Logging automatically -- record hyperparameters, metrics, and system info for every run
    • Comparing runs -- view performance across experiments in tables and charts
    • Reproducing results -- store the exact configuration needed to recreate any run
    • Collaborating -- share dashboards and results with your team

    Weights & Biases (W&B)¶

    Weights & Biases is a hosted experiment tracking platform. It provides dashboards, hyperparameter sweep orchestration, and artifact versioning.

    Setup¶

    Install W&B in your Conda environment or venv:

    $ pip install wandb

    API Key Configuration¶

    W&B requires an API key to authenticate. Get your key from wandb.ai/authorize, then configure it so batch jobs can authenticate without interactive login.

    Store your key in a file readable only by you:

    $ echo "YOUR_API_KEY" > ~/.wandb_api_key
$ chmod 600 ~/.wandb_api_key

    Reference it in your job scripts:

    export WANDB_API_KEY=$(cat ~/.wandb_api_key)

    Warning

    Never hardcode your API key in job scripts or source code. Use the file-based approach above or set the key in your ~/.bashrc.

    Compute nodes have outbound internet

    AICR compute nodes can reach external services (W&B, HuggingFace, PyPI, etc.) but do not accept inbound connections from outside the cluster.

    Logging Metrics in a Training Script¶

    import wandb

wandb.init(
    project="my_project",
    config={
        "learning_rate": 1e-4,
        "batch_size": 32,
        "epochs": 50,
        "architecture": "resnet50",
    },
)

for epoch in range(50):
    train_loss = train_one_epoch(model, train_loader)
    val_loss, val_acc = evaluate(model, val_loader)

    wandb.log({
        "epoch": epoch,
        "train/loss": train_loss,
        "val/loss": val_loss,
        "val/accuracy": val_acc,
    })

wandb.finish()

    Batch Job with W&B¶

    #!/bin/bash
#SBATCH --job-name=train_wandb
#SBATCH --partition=rtx-batch
#SBATCH --gpus=1
#SBATCH --cpus-per-task=8
#SBATCH --mem=32G
#SBATCH --time=08:00:00
#SBATCH --account=ACCOUNT_NAME
#SBATCH --output=%x-%j.out

module purge
module load cuda

eval "$(conda shell.bash hook)"
conda activate myenv

export WANDB_API_KEY=$(cat ~/.wandb_api_key)

python train.py

    Viewing Results¶

    After your job runs, view results in the W&B web dashboard at wandb.ai. The dashboard shows metric plots, system utilization, hyperparameter comparisons, and run logs.

    W&B Offline Mode¶

    If you prefer not to send data from compute nodes (or if the network is temporarily unavailable), use offline mode. W&B writes all logs locally, and you sync them from a login node afterward:

    export WANDB_MODE=offline

    After the job completes, sync from a login node:

    $ wandb sync /scratch/$USER/wandb/offline-run-*

    Tip

    Set WANDB_DIR=/scratch/$USER/wandb in your job script to write W&B logs to scratch instead of your home directory. W&B run directories can grow large.

    W&B Sweeps with Job Arrays¶

    Combine W&B sweeps with Slurm job arrays for hyperparameter searches:

    1. Create a sweep configuration:

      # sweep.yaml
program: train.py
method: bayes
metric:
  name: val/loss
  goal: minimize
parameters:
  learning_rate:
    min: 0.00001
    max: 0.01
  batch_size:
    values: [16, 32, 64, 128]
  dropout:
    min: 0.1
    max: 0.5

    2. Initialize the sweep from a login node:

      $ wandb sweep sweep.yaml

      This prints a sweep ID like USERNAME/PROJECT/SWEEP_ID.

    3. Submit a job array where each task runs a sweep agent:

      #!/bin/bash
#SBATCH --job-name=wandb_sweep
#SBATCH --partition=rtx-batch
#SBATCH --gpus=1
#SBATCH --cpus-per-task=8
#SBATCH --mem=32G
#SBATCH --time=04:00:00
#SBATCH --account=ACCOUNT_NAME
#SBATCH --array=0-9
#SBATCH --output=%x-%a-%j.out

module purge
module load cuda

eval "$(conda shell.bash hook)"
conda activate myenv

export WANDB_API_KEY=$(cat ~/.wandb_api_key)

wandb agent USERNAME/PROJECT/SWEEP_ID

    Each array task launches a sweep agent that pulls configurations from the W&B server and runs trials until the sweep is complete or the job time expires.

    MLflow¶

    MLflow is an open-source experiment tracking tool that can run entirely locally without a hosted service.

    Local Tracking Server¶

    Start an MLflow tracking server on scratch:

    $ pip install mlflow
$ mlflow server \
    --backend-store-uri sqlite:///scratch/$USER/mlflow/mlflow.db \
    --default-artifact-root /scratch/$USER/mlflow/artifacts \
    --host 0.0.0.0 \
    --port 5000

    Access the MLflow UI through an SSH tunnel:

    $ ssh -L 5000:NODE_NAME:5000 USERNAME@login.aicr.ai

    Then open http://localhost:5000 in your browser.

    Logging with MLflow¶

    import mlflow

mlflow.set_tracking_uri("http://localhost:5000")
mlflow.set_experiment("my_experiment")

with mlflow.start_run():
    mlflow.log_params({
        "learning_rate": 1e-4,
        "batch_size": 32,
    })

    for epoch in range(50):
        train_loss = train_one_epoch(model, train_loader)
        mlflow.log_metric("train_loss", train_loss, step=epoch)

    mlflow.log_artifact("model.pt")

    Note

    MLflow does not require internet access. All data stays on the cluster, which may be preferred for sensitive research data.

    TensorBoard¶

    TensorBoard provides visualization for training runs. It works with PyTorch, TensorFlow, and JAX.

    Writing TensorBoard Logs¶

    With PyTorch:

    import os
from torch.utils.tensorboard import SummaryWriter

log_dir = os.path.expandvars("/scratch/$USER/tb_logs/run_01")
writer = SummaryWriter(log_dir=log_dir)

for epoch in range(50):
    train_loss = train_one_epoch(model, train_loader)
    writer.add_scalar("train/loss", train_loss, epoch)
    writer.add_scalar("val/accuracy", val_acc, epoch)

writer.close()

    Viewing TensorBoard via SSH Tunnel¶

    Start TensorBoard on a login node or inside an interactive job:

    $ tensorboard --logdir=/scratch/$USER/tb_logs --port=6006 --bind_all

    Set up an SSH tunnel from your local machine:

    $ ssh -L 6006:NODE_NAME:6006 USERNAME@login.aicr.ai

    Open http://localhost:6006 in your browser.

    Tip

    Point --logdir at a parent directory containing multiple run subdirectories. TensorBoard will display all runs together for easy comparison.

    Storage Best Practices¶

    Experiment logs, checkpoints, and artifacts can consume significant space. Follow these guidelines:

    • Write logs to scratch, not home -- set WANDB_DIR, MLflow artifact roots, and TensorBoard log_dir to paths under /scratch/$USER/
    • Clean up finished experiments -- delete logs for runs you no longer need
    • Archive important results to work -- copy final metrics and key artifacts to /work/INSTITUTION/ for longer-term storage
    export WANDB_DIR=/scratch/$USER/wandb
export MLFLOW_TRACKING_URI=sqlite:///scratch/$USER/mlflow/mlflow.db

    Warning

    Scratch is subject to regular purging. After your experiments are complete, copy important logs and artifacts to /work/INSTITUTION/ or off-cluster before they are removed.
