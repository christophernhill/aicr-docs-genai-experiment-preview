    AI/ML Data Transfer PyTorch Storage

    Working with Large Datasets¶

    Training on large datasets requires careful data staging, efficient I/O, and the right storage tier. AICR's all-flash VAST storage with NFS-RDMA provides high throughput, but how you organize and load your data still matters.

    Storage Tiers for Large Data¶

    Storage Path Quota Best For
    Apps /apps/datasets/, /apps/models/ Shared Cluster-wide shared datasets and pre-trained models (read-only for users)
    Scratch /scratch/$USER/ 10 TB Active training data, checkpoints, temporary files
    Work /work/INSTITUTION/ Varies Institution-level shared datasets, curated data, collaboration
    Home /home/$USER/ 100 GB Code and configs only — too small for datasets
    • Apps (/apps) is a cluster-wide shared directory on VAST storage. Pre-staged datasets and models are available here for all users. Check /apps/datasets/ and /apps/models/ before downloading your own copy.
    • Scratch is where your active training data should live. It is fast, large, and designed for job I/O. Scratch is subject to regular purging.
    • Work is for shared, longer-term project data. Store curated datasets here so teammates can access them without duplicating copies in individual scratch directories.

    Warning

    Do not store large datasets in your home directory. The 100 GB quota fills quickly and the home filesystem is not designed for high-throughput data loading.

    Data Staging Workflow¶

    The recommended workflow for large datasets follows a stage-compute-archive pattern:

    Transfer --> Scratch --> Compute --> Results --> Work

    1. Transfer Data to Scratch¶

    Move your dataset onto the cluster. For large transfers, use Globus or rsync:

    $ rsync -avP /local/path/to/dataset/ USERNAME@transfer.aicr.ai:/scratch/$USER/datasets/my_dataset/

    For datasets from remote repositories:

    $ cd /scratch/$USER/datasets
$ wget https://example.com/dataset.tar.gz
$ tar -xzf dataset.tar.gz
$ rm dataset.tar.gz

    2. Compute from Scratch¶

    Point your training scripts at the scratch path:

    #!/bin/bash
#SBATCH --job-name=train_large
#SBATCH --partition=b200-batch
#SBATCH --gpus=4
#SBATCH --cpus-per-task=32
#SBATCH --mem=128G
#SBATCH --time=24:00:00
#SBATCH --account=ACCOUNT_NAME
#SBATCH --output=%x-%j.out

module purge
module load cuda

eval "$(conda shell.bash hook)"
conda activate myenv

python train.py --data-dir /scratch/$USER/datasets/my_dataset

    3. Archive Results to Work¶

    After training completes, copy important outputs to work for longer-term storage:

    $ cp -r /scratch/$USER/results/experiment_01 /work/INSTITUTION/PROJECT/results/

    Tip

    Scratch is purged regularly. Do not rely on it for long-term storage. Move final models, metrics, and key artifacts to /work/INSTITUTION/ or transfer them off-cluster.

    NFS-RDMA Performance¶

    AICR uses VAST all-flash storage with NFS-RDMA, providing high-bandwidth access from every compute node. To get the best performance:

    • Use multiple parallel connections -- the storage system scales with concurrent I/O. Using 16 parallel data loading workers can saturate the available bandwidth far better than a single-threaded reader.
    • Read large sequential blocks -- formats that store data contiguously (binary files, memory-mapped arrays, sharded archives) outperform millions of tiny individual files.
    • Avoid many small files -- directories with hundreds of thousands of small files (e.g., one PNG per sample) create metadata bottlenecks. Use sharded formats instead.

    Note

    With 16 parallel connections per node, the NFS-RDMA fabric delivers high aggregate throughput. Structure your data pipeline to take advantage of this concurrency.

    PyTorch DataLoader Optimization¶

    The PyTorch DataLoader is the primary interface for feeding data to GPU training. Tuning its parameters can dramatically improve throughput.

    Key Parameters¶

    from torch.utils.data import DataLoader

train_loader = DataLoader(
    dataset,
    batch_size=64,
    shuffle=True,
    num_workers=8,         # parallel data loading processes
    pin_memory=True,       # use pinned (page-locked) memory for faster GPU transfer
    prefetch_factor=2,     # batches prefetched per worker
    persistent_workers=True,  # keep workers alive between epochs
)
    Parameter Recommended Value Why
    num_workers 4--8 per GPU Parallel data loading saturates I/O; match to --cpus-per-task
    pin_memory True Enables async CPU-to-GPU memory transfer
    prefetch_factor 2--4 Prefetches batches so the GPU never waits on data
    persistent_workers True Avoids worker respawn overhead between epochs

    Tip

    A good starting point is num_workers = 4 * NUM_GPUS. If you have --cpus-per-task=32 and 4 GPUs, set num_workers=8 per DataLoader. Monitor GPU utilization with nvidia-smi -- if GPUs are idle between steps, increase num_workers or prefetch_factor.

    Example Batch Job¶

    #!/bin/bash
#SBATCH --job-name=train_dataloader
#SBATCH --partition=rtx-batch
#SBATCH --gpus=4
#SBATCH --cpus-per-task=32
#SBATCH --mem=128G
#SBATCH --time=12:00:00
#SBATCH --account=ACCOUNT_NAME
#SBATCH --output=%x-%j.out

module purge
module load cuda

eval "$(conda shell.bash hook)"
conda activate myenv

python train.py \
    --data-dir /scratch/$USER/datasets/imagenet \
    --num-workers 8 \
    --batch-size 64

    HuggingFace Streaming Mode¶

    For datasets too large to download entirely, or when you want to start training immediately, use HuggingFace's streaming mode:

    from datasets import load_dataset

dataset = load_dataset(
    "allenai/c4",
    "en",
    split="train",
    streaming=True,
)

for example in dataset:
    # Process examples one at a time -- no full download needed
    tokens = tokenizer(example["text"])

    Streaming mode:

    • Downloads data on the fly, so you can begin training immediately
    • Uses minimal disk space -- no need to store the full dataset
    • Works with IterableDataset and PyTorch DataLoader
    from torch.utils.data import DataLoader

torch_dataset = dataset.with_format("torch")
loader = DataLoader(torch_dataset, batch_size=32)

    Note

    Streaming requires network access from compute nodes. AICR compute nodes have outbound internet, so streaming works directly from jobs.

    Sharded Dataset Formats¶

    For very large datasets (hundreds of GB to TB scale), sharded formats are more efficient than directories of individual files. They reduce metadata overhead and enable parallel reading.

    WebDataset¶

    WebDataset stores data in tar archives (shards) that can be read sequentially at high speed:

    import os
import webdataset as wds

shard_path = os.path.expandvars("/scratch/$USER/datasets/imagenet-shards/shard-{000000..001000}.tar")
dataset = (
    wds.WebDataset(shard_path)
    .decode("pil")
    .to_tuple("jpg", "cls")
    .batched(64)
)

loader = wds.WebLoader(dataset, num_workers=8, batch_size=None)

    Mosaic StreamingDataset¶

    Mosaic StreamingDataset writes data in an optimized binary format and supports deterministic, resumable streaming:

    import os
from streaming import StreamingDataset

dataset = StreamingDataset(
    local=os.path.expandvars("/scratch/$USER/datasets/my_mosaic_dataset"),
    shuffle=True,
)

    Creating Sharded Datasets¶

    Convert your dataset to shards before training, not during:

    $ python create_shards.py \
    --input /scratch/$USER/datasets/raw_images/ \
    --output /scratch/$USER/datasets/shards/ \
    --shard-size 1GB

    Tip

    Shard sizes of 256 MB to 1 GB generally work well. Larger shards reduce metadata overhead; smaller shards improve shuffle quality and enable more parallelism.

    Shared Dataset Repository¶

    If multiple users in your project need the same dataset, store a single copy in the shared work directory rather than duplicating it across individual scratch directories:

    /work/INSTITUTION/datasets/
    imagenet/
    c4/
    openwebtext/

    Set appropriate group permissions so all project members can read:

    $ chmod -R g+rX /work/INSTITUTION/datasets/my_dataset

    Note

    Coordinate with your team to decide who prepares and maintains shared datasets. A single curated copy in /work/INSTITUTION/datasets/ saves storage and avoids version inconsistencies.
