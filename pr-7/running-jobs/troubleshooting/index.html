    Running Jobs Slurm Troubleshooting

    Job Troubleshooting¶

    Common job problems and how to fix them.

    Common Failures¶

    Problem Likely Cause Solution
    Job stays PENDING Resources unavailable or fairshare is low Run squeue -j JOBID to see the reason code. Reduce resource requests or wait for fairshare recovery.
    PENDING reason: Priority Other jobs have higher priority Your fairshare may be temporarily low. Wait, or reduce your resource request.
    PENDING reason: Resources Not enough nodes/GPUs available Check sinfo -p PARTITION for capacity. Reduce --nodes or --gpus, or try a different partition.
    PENDING reason: QOSMaxJobsPerUserLimit Too many devel jobs Devel partitions are limited to 4 concurrent jobs. Wait for one to finish or cancel one.
    error: Batch job submission failed: Invalid account Wrong --account value Use your institution's account: bu, hu, mit, neu, umass, uml, umb, umassd, umassmed, yale, ma, aihub. See Requesting Access for the account matching your institution.
    error: Unable to allocate resources: Invalid partition Partition name misspelled Valid partitions: cpu, rtx-batch, rtx-devel, b200-batch, b200-devel.
    Job killed: OUT_OF_MEMORY Program exceeded requested memory Increase --mem or --mem-per-cpu. Check actual usage with sacct -j JOBID --format=MaxRSS.
    Job killed: TIMEOUT Wall time too short Increase --time. Check how long similar jobs took with sacct. Consider checkpointing.
    FAILED with exit code 1 Your program crashed Check the output and error files. Debug interactively: salloc --partition=rtx-devel --gpus=1 --cpus-per-task=4 --mem=16G --time=01:00:00 --account=ACCOUNT_NAME
    GPUs not visible (nvidia-smi shows nothing) Missing --gpus or wrong partition Verify you requested GPUs (--gpus=N or --gres=gpu:N) and submitted to a GPU partition.
    CUDA error: no CUDA-capable device GPU not allocated or CUDA not loaded Check nvidia-smi first. If GPUs show up, load the CUDA module: module load cuda.
    Node failure (NODE_FAIL state) Hardware issue on the compute node Not your fault. Resubmit the job — Slurm will assign a different node.

    Debugging Strategies¶

    Check Output Files¶

    The first thing to do when a job fails is read the output and error files:

    $ cat JOBNAME-JOBID.out
$ cat JOBNAME-JOBID.err

    Debug Interactively¶

    Request a short interactive session on the same partition and reproduce the issue:

    $ salloc --partition=rtx-devel --gpus=1 --cpus-per-task=4 --mem=16G --time=01:00:00 --account=ACCOUNT_NAME

    Then run your commands step by step to isolate the failure.

    Check Resource Usage¶

    If your job completed but the results seem wrong, check what resources it actually used:

    $ sacct -j JOBID --format=JobID,Elapsed,MaxRSS,MaxVMSize,State,ExitCode

    If MaxRSS is close to your --mem request, the job may have been silently constrained. Request more memory.

    Check Module Environment¶

    If software behaves differently in a batch job than interactively, ensure your job script starts with module purge and explicitly loads everything it needs:

    module purge
module load cuda python

    Do not rely on modules loaded in your login session — batch jobs start with a clean environment.

    Getting More Help¶

    If you cannot resolve the issue:

    1. Note the job ID, partition, error message, and what you tried
    2. Contact your institution's RC team — see Getting Help
