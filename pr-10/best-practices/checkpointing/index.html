    Checkpointing Long Jobs¶

    Jobs on AICR have wall-time limits (24 hours on batch partitions, 4 hours on devel). Training runs that exceed these limits need checkpointing -- periodically saving the full training state so the job can resume from where it left off.

    Why Checkpoint¶

    • Time limits. Slurm kills jobs that reach their --time limit. Without checkpoints, all progress is lost.
    • Preemption. If your job is preempted by a higher-priority job, a recent checkpoint lets you restart without losing work.
    • Hardware failures. Nodes can go down. Checkpoints are your safety net.
    • Iterative development. Save at key milestones so you can branch experiments from intermediate states.

    General Strategy¶

    1. Save a checkpoint every N epochs or N steps.
    2. Write checkpoints to /scratch/$USER (fast I/O, large quota).
    3. Keep at least the two most recent checkpoints (in case the latest is corrupted).
    4. On restart, detect the latest checkpoint and resume from it.

    Tip

    Save checkpoints at intervals that balance I/O cost against lost work. For a 24-hour job, checkpointing every 1-2 hours is a reasonable starting point.

    Note

    The Python examples below use /scratch/$USER/PROJECT_NAME/ as placeholder paths. In your actual code, use os.path.expandvars() or os.environ["USER"] to expand $USER, since shell variables don't expand in Python strings.

    PyTorch¶

    Saving a Checkpoint¶

    Save the model, optimizer, scheduler, epoch, and any other state needed to resume:

    import torch

def save_checkpoint(model, optimizer, scheduler, epoch, loss, path):
    torch.save({
        'epoch': epoch,
        'model_state_dict': model.state_dict(),
        'optimizer_state_dict': optimizer.state_dict(),
        'scheduler_state_dict': scheduler.state_dict(),
        'loss': loss,
    }, path)

# In your training loop
save_checkpoint(model, optimizer, scheduler, epoch, loss,
                f'/scratch/$USER/PROJECT_NAME/checkpoint_epoch_{epoch}.pt')

    Loading a Checkpoint¶

    def load_checkpoint(path, model, optimizer, scheduler):
    checkpoint = torch.load(path, weights_only=False)
    model.load_state_dict(checkpoint['model_state_dict'])
    optimizer.load_state_dict(checkpoint['optimizer_state_dict'])
    scheduler.load_state_dict(checkpoint['scheduler_state_dict'])
    return checkpoint['epoch'], checkpoint['loss']

# At the start of training
import os
CHECKPOINT_PATH = '/scratch/$USER/PROJECT_NAME/checkpoint_latest.pt'

start_epoch = 0
if os.path.exists(CHECKPOINT_PATH):
    start_epoch, _ = load_checkpoint(CHECKPOINT_PATH, model, optimizer, scheduler)
    start_epoch += 1  # resume from next epoch
    print(f'Resuming from epoch {start_epoch}')

for epoch in range(start_epoch, NUM_EPOCHS):
    train_one_epoch(model, optimizer, dataloader)
    save_checkpoint(model, optimizer, scheduler, epoch, loss, CHECKPOINT_PATH)

    PyTorch Lightning¶

    PyTorch Lightning has built-in checkpointing:

    from pytorch_lightning.callbacks import ModelCheckpoint

checkpoint_callback = ModelCheckpoint(
    dirpath='/scratch/$USER/PROJECT_NAME/checkpoints',
    filename='model-{epoch:02d}-{val_loss:.2f}',
    save_top_k=2,
    monitor='val_loss',
)

trainer = pl.Trainer(
    max_epochs=100,
    callbacks=[checkpoint_callback],
)

# Resume from checkpoint
trainer.fit(model, datamodule=dm, ckpt_path='/scratch/$USER/PROJECT_NAME/checkpoints/CHECKPOINT_FILE')

    TensorFlow / Keras¶

    Use the ModelCheckpoint callback:

    import tensorflow as tf

checkpoint_callback = tf.keras.callbacks.ModelCheckpoint(
    filepath='/scratch/$USER/PROJECT_NAME/checkpoints/model-{epoch:02d}.keras',
    save_freq='epoch',
    save_best_only=False,
)

model.fit(
    train_dataset,
    epochs=100,
    initial_epoch=RESUME_EPOCH,
    callbacks=[checkpoint_callback],
)

    To resume, load the model and set initial_epoch:

    model = tf.keras.models.load_model('/scratch/$USER/PROJECT_NAME/checkpoints/model-50.keras')
model.fit(train_dataset, epochs=100, initial_epoch=50, callbacks=[checkpoint_callback])

    Framework-Agnostic: DMTCP¶

    DMTCP (Distributed MultiThreaded CheckPointing) can checkpoint arbitrary Linux processes, including applications without built-in checkpoint support. It works by saving the entire process state (memory, open files, network connections) to disk.

    $ dmtcp_launch ./my_application
$ dmtcp_command --checkpoint          # trigger a checkpoint
$ dmtcp_restart ckpt_*.dmtcp          # restart from checkpoint

    Note

    DMTCP checkpoints tend to be large (they save all process memory) and may not work with all GPU libraries. Use application-level checkpointing (PyTorch, TensorFlow) when available. DMTCP is best for legacy codes or tools that lack native checkpoint support.

    Slurm Integration¶

    Setting Time Limits¶

    Set --time to allow enough time for at least one checkpoint cycle. If your job checkpoints every 2 hours, request at least 2.5 hours so there is time for the final checkpoint before the job is killed:

    #SBATCH --time=12:00:00
#SBATCH --account=ACCOUNT_NAME

    Signal-Based Checkpointing¶

    Slurm can send a signal to your job before killing it, giving your code time to save a final checkpoint. Use --signal to request a warning:

    #SBATCH --signal=B:USR1@300

    This sends SIGUSR1 to your job 300 seconds (5 minutes) before the time limit. Your training script can catch this signal and trigger a checkpoint:

    import signal
import sys

def handle_signal(signum, frame):
    print('Received signal, saving checkpoint...')
    save_checkpoint(model, optimizer, scheduler, epoch, loss, CHECKPOINT_PATH)
    sys.exit(0)

signal.signal(signal.SIGUSR1, handle_signal)

    Automatic Requeue¶

    Combine checkpointing with --requeue to chain jobs automatically. If a job is preempted or times out, Slurm requeues it and the restarted job picks up from the latest checkpoint:

    #!/bin/bash
#SBATCH --job-name=long-training
#SBATCH --partition=rtx-batch
#SBATCH --gpus=1
#SBATCH --mem=32G
#SBATCH --time=12:00:00
#SBATCH --account=ACCOUNT_NAME
#SBATCH --signal=B:USR1@300
#SBATCH --requeue

CHECKPOINT_DIR=/scratch/$USER/PROJECT_NAME/checkpoints

python train.py \
  --checkpoint-dir $CHECKPOINT_DIR \
  --auto-resume

    Job Chaining with Dependencies¶

    For finer control, chain multiple jobs so the next one starts after the previous one finishes:

    $ JOB1=$(sbatch --parsable job.sh)
$ sbatch --dependency=afterany:$JOB1 job.sh

    Each job in the chain loads the checkpoint from the previous run.

    Checkpoint Housekeeping¶

    Checkpoints can consume significant scratch space, especially for large models. Keep your checkpoint directory clean:

    # Keep only the 2 most recent checkpoints
$ ls -t /scratch/$USER/PROJECT_NAME/checkpoints/*.pt | tail -n +3 | xargs rm -f

    Build cleanup into your training script or include it at the end of your Slurm job.
