    Best Practices Performance Running Jobs Slurm

    Efficient Resource Usage¶

    Requesting only the resources your job needs improves your scheduling priority, reduces wait times, and lets the cluster serve more researchers. This page covers how to right-size requests, verify actual usage, and understand the fairshare system.

    Request What You Need¶

    Over-requesting resources (more GPUs, CPUs, or memory than your code uses) wastes capacity and lowers your fairshare priority. Start with a modest request, measure actual usage, and adjust.

    Memory¶

    AICR sets DefMemPerCPU=1000 (1 GB per CPU). If you request --cpus-per-task=4 without specifying --mem, you get 4 GB automatically. For jobs that need more, specify memory explicitly:

    #SBATCH --mem=32G

    Tip

    Run a short test job first, then check MaxRSS with sacct to find your actual peak memory. Request 10-20% above that value for production runs.

    GPUs¶

    Request only the GPUs your code can use. A single-GPU training script does not benefit from --gpus=2. If you are unsure whether your code supports multi-GPU, start with one:

    #SBATCH --gpus=1

    Wall Time¶

    Estimate how long your job needs and add a buffer. Shorter time limits improve scheduling priority:

    #SBATCH --time=04:00:00

    Warning

    Jobs that reach their time limit are killed. If your job might exceed the limit, implement checkpointing to save progress.

    Verify Usage with sacct¶

    After a job completes, check what it actually consumed:

    $ sacct -j JOBID --format=JobID,JobName,Elapsed,MaxRSS,AllocCPUS,AllocTRES,State

    Compare MaxRSS (peak memory) against your --mem request and Elapsed against your --time limit. If the gap is large, reduce your request for the next run.

    Check your recent job history:

    $ sacct -u $USER --starttime=now-7days --format=JobID,JobName,Elapsed,MaxRSS,State

    What to Look For¶

    Field Compare Against Action If Over-Requested
    MaxRSS --mem Lower memory request
    Elapsed --time Shorten time limit
    AllocCPUS CPU utilization Request fewer CPUs
    AllocTRES GPU utilization Request fewer GPUs

    Monitor GPU Utilization¶

    For running GPU jobs, check whether your GPUs are actually being used:

    $ ssh NODE_NAME nvidia-smi

    Look at the GPU-Util column. If it is consistently below 30%, your code may not be fully utilizing the GPU. Common causes include data loading bottlenecks, small batch sizes, or CPU-bound preprocessing.

    Note

    You can only SSH to compute nodes where you have an active Slurm job. Find your node name with squeue -u $USER.

    Understanding Fairshare¶

    AICR uses Slurm's fairshare algorithm to prioritize jobs across a three-level hierarchy: Institution → Project → User. All institutions have equal shares. The more resources you consume relative to your institution's and project's allocation, the lower your priority becomes. Priority recovers over time as your usage ages out.

    Fairshare Structure¶

    AICR has 8 top-level accounts representing the participating institutions and entities:

    Account Institution
    bu Boston University
    hu Harvard University
    mit Massachusetts Institute of Technology
    neu Northeastern University
    umass University of Massachusetts (parent)
    yale Yale University
    ma Commonwealth of Massachusetts
    aihub Massachusetts AI Hub

    The UMass account has 5 sub-accounts corresponding to its campuses:

    Sub-Account Campus
    uml UMass Lowell
    umb UMass Boston
    umassd UMass Dartmouth
    umassmed UMass Chan Medical School
    umass UMass Amherst

    Always specify the correct account in your jobs:

    #SBATCH --account=ACCOUNT_NAME

    Check Your Priority¶

    $ sprio -u $USER

    $ sshare -u $USER

    How to Maintain Good Priority¶

    1. Request only what you need -- over-requesting wastes your fairshare budget even if the resources sit idle.
    2. Use devel partitions for testing -- short test runs on rtx-devel or b200-devel consume less fairshare than running on batch partitions.
    3. Cancel jobs you no longer need -- a pending or running job you forgot about still affects scheduling.
    4. Check actual usage -- use sacct after each job and adjust future requests.

    Example: Right-Sizing a Job¶

    A researcher submits a PyTorch training job and discovers after the first run that it uses 12 GB of memory and takes 2.5 hours:

    Before (over-requested):

    #SBATCH --mem=64G
#SBATCH --cpus-per-task=16
#SBATCH --gpus=2
#SBATCH --time=12:00:00
#SBATCH --account=ACCOUNT_NAME

    After (right-sized):

    #SBATCH --mem=16G
#SBATCH --cpus-per-task=4
#SBATCH --gpus=1
#SBATCH --time=03:30:00
#SBATCH --account=ACCOUNT_NAME

    The right-sized job uses a fraction of the resources, improving fairshare priority and reducing queue wait time.
