    Data Management¶

    AICR storage is designed for active computation, not long-term archival. This page describes the recommended workflow for moving data through the system efficiently and keeping storage tidy for everyone.

    The Data Workflow: Stage, Compute, Copy, Clean¶

    Follow this four-step pattern for every project:

    Stage --> Compute --> Copy --> Clean

    1. Stage¶

    Transfer your input data to AICR before submitting jobs. Use /scratch/$USER for large datasets that will be consumed by running jobs:

    $ rsync -avz LOCAL_DATA_DIR/ transfer.aicr.ai:/scratch/$USER/PROJECT_NAME/input/

    For shared datasets that multiple team members need, stage to your institution's work directory:

    $ rsync -avz SHARED_DATASET/ transfer.aicr.ai:/work/INSTITUTION/PROJECT_NAME/data/

    Tip

    For large transfers (over 100 GB), use Globus for faster, resumable transfers. See Transferring Data.

    2. Compute¶

    Point your Slurm jobs at the staged data. Read input from scratch or work, and write output (checkpoints, results, logs) to scratch:

    #!/bin/bash
#SBATCH --job-name=training
#SBATCH --partition=rtx-batch
#SBATCH --gpus=1
#SBATCH --mem=32G
#SBATCH --time=08:00:00
#SBATCH --account=ACCOUNT_NAME

INPUT_DIR=/scratch/$USER/PROJECT_NAME/input
OUTPUT_DIR=/scratch/$USER/PROJECT_NAME/output

mkdir -p $OUTPUT_DIR

python train.py \
  --data-dir $INPUT_DIR \
  --output-dir $OUTPUT_DIR

    3. Copy¶

    After jobs complete, copy results you want to keep off AICR. Transfer final models, key results, or publications data to your local system or institutional storage:

    $ rsync -avz transfer.aicr.ai:/scratch/$USER/PROJECT_NAME/output/final_model/ LOCAL_RESULTS_DIR/

    For results that collaborators need on AICR, move them to work:

    $ mv /scratch/$USER/PROJECT_NAME/output/shared_results /work/INSTITUTION/PROJECT_NAME/results/

    4. Clean¶

    Remove data from scratch once you no longer need it. Do not leave stale data sitting in scratch:

    $ rm -rf /scratch/$USER/PROJECT_NAME

    Danger

    Scratch is subject to regular purging. Files that have not been accessed recently will be deleted automatically. Never rely on scratch for data you cannot regenerate.

    Choosing the Right Storage¶

    Storage Path Best For Not For
    Home /home/$USER Code, scripts, configs, small conda envs Large datasets, job output
    Scratch /scratch/$USER Active job I/O, checkpoints, temp data Long-term storage, irreplaceable data
    Work /work/INSTITUTION Shared project data, reference datasets Personal files, job temp files

    Home (/home/$USER)¶

    • Quota: 100 GB
    • Snapshots: 7-day
    • Purpose: Source code, job scripts, configuration files, small software environments.
    • Keep your home directory lean. Large conda environments or datasets do not belong here.

    Scratch (/scratch/$USER)¶

    • Quota: 10 TB
    • Snapshots: None
    • Purge: Regular (files not accessed recently are removed)
    • Purpose: Data actively being read or written by running jobs. Checkpoints, intermediate results, large model files during training.

    Warning

    Scratch has no snapshots and no backups. If a file is deleted (by you or by purge), it is gone.

    Work (/work/INSTITUTION)¶

    • Quota: Varies by institution
    • Snapshots: 7-day
    • Purpose: Shared datasets, reference data, and collaboration space. Data here persists longer than scratch but should still be cleaned up when a project ends.

    Checking Your Usage¶

    Monitor your quota to avoid failed jobs caused by full filesystems:

    $ df -h /home/$USER
$ df -h /scratch/$USER
$ du -sh /home/$USER/*

    See Managing Quotas for detailed quota commands.

    Tips for Large Datasets¶

    • Avoid many small files. Thousands of tiny files strain the filesystem. Archive them into tar files or use formats like HDF5, TFRecord, or WebDataset.
    • Compress when possible. Use tar -czf for datasets that compress well. Decompress to scratch before jobs run.
    • Stage once, use many times. If multiple jobs read the same dataset, stage it once to scratch and point all jobs at the same path.
    • Use symlinks sparingly. Symlinks across filesystems (e.g., from home to scratch) can break when scratch is purged.

    Regular Maintenance¶

    Build cleanup into your workflow:

    # Check what you have in scratch
$ du -sh /scratch/$USER/*/

# Remove completed project data
$ rm -rf /scratch/$USER/OLD_PROJECT

# Check home usage
$ du -sh /home/$USER

    Tip

    Set a calendar reminder to review your scratch and work directories monthly. Projects that finish often leave behind hundreds of gigabytes of checkpoints and intermediate data.
