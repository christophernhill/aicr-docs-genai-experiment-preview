    Glossary¶

    Definitions of HPC and AI terms used throughout this documentation. Terms are listed alphabetically.

    Account (Slurm)
    An organizational unit in Slurm tied to your institution. You specify it with --account when submitting jobs. AICR accounts include bu, hu, mit, neu, umass, uml, umb, umassd, umassmed, yale, ma, and aihub.
    Apptainer
    A container runtime for HPC (formerly Singularity). Lets you package an entire software environment into a single .sif file and run it on the cluster without root privileges. See Containers.
    Batch Job
    A job submitted with sbatch that runs unattended. You write a job script, submit it, and collect the results later. Contrast with an interactive job.
    Checkpoint
    A saved snapshot of a running job's state (model weights, optimizer state, epoch number). Allows long-running jobs to resume after a timeout or failure. See Checkpointing.
    Compute Node
    A machine in the cluster where your jobs actually run. You do not log into compute nodes directly — the scheduler assigns them to your jobs.
    Conda
    A package and environment manager commonly used for Python and R. Lets you create isolated environments with specific package versions. See Python and Conda.
    Container
    A packaged software environment that includes an OS, libraries, and applications in a single file. On AICR, containers are run with Apptainer. See Containers.
    CPU
    Central Processing Unit. On AICR, the cpu partition provides nodes with 128 cores and 1 TB memory each, intended for data analysis and workflow orchestration.
    CUDA
    NVIDIA's programming platform for GPUs. Required for running GPU-accelerated code. The CUDA toolkit is available as a module on AICR.
    DDP (Distributed Data Parallel)
    A PyTorch strategy for training a model across multiple GPUs. Each GPU holds a copy of the model and processes a different data shard. Gradients are synchronized after each step. See Distributed Training.
    Fairshare
    Slurm's mechanism for allocating resources fairly across a three-level hierarchy: Institution → Project → User. If your institution and project have used less than their share recently, your jobs get higher priority. You do not need to manage this yourself.
    FSDP (Fully Sharded Data Parallel)
    A PyTorch strategy for training models that are too large for a single GPU's memory. The model parameters, gradients, and optimizer states are sharded across GPUs. See Distributed Training.
    Globus
    A data transfer service for moving large files between systems. Transfers continue in the background — you do not need to stay logged in. See Transferring Data.
    GPU (Graphics Processing Unit)
    A processor designed for parallel computation. AICR has NVIDIA B200 and RTX PRO 6000 GPUs, 8 per node. Used for training and running AI models. See GPU Overview.
    GRES (Generic Resources)
    Slurm's mechanism for requesting specialized resources like GPUs. You can request GPUs with --gpus=2 or --gres=gpu:2 (both are equivalent).
    Home Directory
    Your personal storage at /home/USERNAME. 100 GiB quota with 7-day snapshots. For code, configs, and small files. See Filesystem Overview.
    InfiniBand
    A high-speed network connecting AICR nodes. NDR400 InfiniBand provides 400 Gb/s bandwidth for GPU-to-GPU communication and storage access.
    Interactive Job
    A job started with salloc that gives you a shell on a compute node for real-time development and testing. Limited to 4 concurrent jobs on devel partitions.
    Job
    A unit of work submitted to the scheduler. A job specifies what resources you need (CPUs, GPUs, memory, time) and what to run. See Slurm Basics.
    Job Array
    A way to submit many similar jobs from a single script. Each task gets a unique ID via $SLURM_ARRAY_TASK_ID. Useful for parameter sweeps. See Job Arrays.
    Job Script
    A bash script containing #SBATCH directives (resource requests) and the commands to run. Submitted with sbatch.
    Login Node
    The machine you connect to when you SSH into the cluster. For editing files, submitting jobs, and light tasks only — not for running computation. Limited to 4 CPU cores and 8 GB memory per user.
    Lmod
    The module system used on AICR. Manages software by modifying your shell environment. See Modules.
    MIG (Multi-Instance GPU)
    An NVIDIA technology that partitions a physical GPU into smaller, isolated instances. May be available on some AICR nodes for lighter workloads.
    Module
    A software package managed by Lmod. Loading a module makes the software available in your session. Commands: module avail, module load, module purge.
    MPI (Message Passing Interface)
    A standard for communication between processes running on different nodes. Used for multi-node parallel jobs. AICR uses PMIx for MPI process management.
    NCCL (NVIDIA Collective Communications Library)
    A library for fast GPU-to-GPU communication. Used by PyTorch DDP and FSDP for gradient synchronization during distributed training.
    Node
    A single machine in the cluster. AICR has GPU nodes (a-series and b-series), CPU nodes (w-series), and service nodes.
    OnDemand
    A web portal for accessing the cluster through your browser. Provides a file manager, terminal, and interactive applications without SSH. See OnDemand.
    Partition
    A group of nodes in Slurm with shared resource limits. AICR has five partitions: cpu, rtx-batch, rtx-devel, b200-batch, b200-devel. You choose a partition with --partition.
    PMIx
    Process Management Interface for Exascale. The process launcher used by AICR's Slurm for MPI jobs. Specify with srun --mpi=pmix.
    QoS (Quality of Service)
    A Slurm mechanism that sets job limits. The interactive QoS on devel partitions limits users to 4 concurrent jobs.
    RDMA (Remote Direct Memory Access)
    A networking technology that transfers data directly between memories without involving the CPU. AICR uses NFS-RDMA over InfiniBand for fast storage access.
    Scratch
    Temporary high-capacity storage at /scratch/USERNAME. ~10 TiB quota with ~15M file limit, no snapshots, subject to regular purging (~30 days). For active job data only. See Filesystem Overview.
    Scheduler
    Software that manages compute resources and runs your jobs on available nodes. AICR uses Slurm 25.11.1.
    Slurm
    The workload manager and job scheduler used on AICR. Commands include sbatch, salloc, srun, squeue, sacct, scancel. See Slurm Basics.
    Snapshot
    A read-only point-in-time copy of your files. Available on Home and Work storage (not Scratch). Used to recover accidentally deleted files. See Restoring from Snapshots.
    SSH (Secure Shell)
    A protocol for connecting to remote computers over a network. The primary way to access AICR from a terminal. AICR uses SSH certificate authentication with short-lived CA-signed certificates. See SSH.
    vLLM
    An open-source inference engine for large language models. Supports multi-GPU serving with tensor parallelism. See Inference and Serving.
    Wall Time
    The maximum real-world time a job is allowed to run, specified with --time. If a job exceeds its wall time, it is terminated.
    Work Directory
    Shared storage for your institution at /work/INSTITUTION/PROJECTNAME. Per-project quota with 7-day snapshots. For project data and collaboration. See Filesystem Overview.
